# leomarameo7.github.io

Source code for Leonardo Capitani's personal/academic website, published at
**<https://leomarameo7.github.io>**.

The site is built with [Quarto](https://quarto.org/) and hosted for free on
GitHub Pages. If you have never worked with a Quarto website before, the
[RLadiesSB Quarto Websites tutorial](https://samanthacsik.github.io/RLadiesSB-quarto-websites/)
is a good, gentle introduction to the same concepts used here (project
config, `.qmd` source files, the output directory, and the render step) —
this README assumes no prior knowledge and explains this specific repo, but
that tutorial is the place to go for the general Quarto mental model.

## The one-paragraph version

You write content in plain-text `.qmd` files (Markdown, optionally with R
code chunks). Running `quarto render` turns every `.qmd` file into a
finished `.html` page and drops it into `docs/`. Nothing in `docs/` is ever
hand-edited — it is fully generated. GitHub Pages is configured to serve the
`docs/` folder of the `main` branch, so pushing a fresh render to `main` is
what actually updates the live site. A GitHub Actions workflow does that
render-and-commit step automatically on every push, so in practice you edit
`.qmd`/config files, push, and the live site catches up on its own within a
minute or two.

### What each source page is

Every top-level `.qmd` file (and every `.qmd` inside `notebooks/` and
`scientific_projects/`) is one page of the website. Quarto turns
`some/path/page.qmd` into `docs/some/path/page.html`, preserving the folder
layout — so the URL structure of the live site mirrors the folder structure
of the source files.

The R notebooks (`notebooks/*/*.qmd`) are a bit more involved than the other
pages: they contain executable R code chunks (fenced with ` ```{r} `).
Rendering one of these doesn't just convert Markdown to HTML — it actually
runs the R code (fitting statistical models, drawing plots, etc.) and embeds
the results in the page.

### `docs/` is generated, not written

This is the single most important thing to know about this repo: **`docs/`
is build output**. Every `.html` file in it, `search.json`, `site_libs/`,
all of it, is produced by running `quarto render`. The only two things
under `docs/` that are *not* generated by Quarto are `docs/media/` (static
images referenced by pages) and `docs/cv.pdf` (compiled separately, see
below) — everything else gets overwritten on every render.

## Rendering the site locally

```bash
quarto render --no-clean
```

Use `--no-clean`, not a plain `quarto render`. By default Quarto empties the
entire output directory before rendering, which would delete `docs/media/`
and `docs/cv.pdf` since those aren't files Quarto itself manages. `--no-clean`
tells it to leave anything it doesn't recognize alone.

To preview changes live while editing, without doing a full render:

```bash
quarto preview
```

## How deployment actually works

1. You push a commit to `main` (typically just source changes — `.qmd`,
   `_quarto.yml`, `styles.scss`, etc.).
2. `.github/workflows/render.yml` picks that push up, installs Quarto/R,
   runs `quarto render --no-clean`, and — if that produced any changes in
   `docs/` — commits and pushes them back to `main` under the
   `github-actions[bot]` user.
3. GitHub Pages serves whatever is currently in `docs/` on `main`, so that
   bot commit is what makes the change visible on the live site.

This workflow is deliberately excluded from triggering itself: it ignores
pushes that only touch `docs/**` or `CV/**` (the CV has its own workflow,
below), so the two automated commits don't loop into each other.

## The CV

`docs/cv.pdf` (linked from the home page) is compiled from `CV/CV.typ`, a
[Typst](https://typst.org/) document, not written directly in this repo day
to day — it's authored on [typst.app](https://typst.app) and synced into
`CV/` via Typst's "Git Sync" feature. Whenever something changes under
`CV/`, `.github/workflows/compile-cv.yml` compiles it straight to
`docs/cv.pdf` and commits the result. You generally shouldn't need to touch
`CV/` by hand — edit on typst.app and push from there.

## Adding a new page

1. Create a new `.qmd` file wherever it logically belongs (top level, or a
   new subfolder under `notebooks/`/`scientific_projects/`).
2. If it should show up in the navbar or sidebar, add it to `_quarto.yml`
   (see the `navbar:`/`sidebar:` sections).
3. If it's a listing page (like `notebooks/index_notebooks.qmd` or
   `scientific_projects/index_sci_pro.qmd`), link the new page from there
   too.
4. Render locally to check it looks right, then commit and push — CI takes
   care of the rest.
